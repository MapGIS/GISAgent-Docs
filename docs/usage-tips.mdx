# 如何更顺手地使用 GIS Agent

本文整理了 GIS Agent 在提示词编写、数据源选择、位置纠偏和问题排查方面的常见做法。

如果你正在问“如何使用 GIS Agent”或“如何描述需求”，优先阅读本页。

***

## 一、提示词写法

与 GIS Agent 协作时，尽量把需求说得具体一些。下面三种写法更容易直接上手。

### 1.1 需求描述模板：数据源 + 操作 + 输出

这是最常用的一种写法。建议按 **“数据源 → 操作 → 输出”** 的顺序描述。

**标准公式**：`使用 [数据源] 进行 [操作]，输出为 [格式]，并可追加 [后续动作]。`

**示例**：

> 使用时空数据库中的 OSM 数据，裁剪出武汉市行政区范围，发布为矢量瓦片服务并保存结果。

**这样写的好处：** 可以减少歧义，也更容易一次说清任务目标。

### 1.2 迭代修正：先描述现象，再继续调整

当输出结果不符合预期时，可以先描述现象，再让 Agent 给出修改建议。

**应用流程**：

1. **现象描述**：直接告诉 Agent “地图上的标注字体太小，看不清楚”。
2. **接受建议**：Agent 会先分析可能原因，再给出调整建议。
3. **执行修正**：根据建议，下达明确的样式修改指令。

这种方式更适合在已有结果的基础上继续调整。

### 1.3 过程存档：保留关键结果

对于耗时较长的分析流程，可以在关键步骤后保留中间成果，便于追溯和复用。

**典型指令**：

> “将当前地图服务发布的所有参数和结果存档。”
> “导出本次空间分析的全过程日志。”

**这样做的好处：** 后续修改、复查或复用时会更方便。

## 二、数据使用：优先明确数据源

**建议**：在执行空间数据处理任务前，尽量明确指定数据源。很多情况下，优先使用时空数据库会更稳定。

### 2.1 为什么要先指定数据源？

* **性能与架构优势**：时空数据库针对海量时空数据做了专门优化，其内置的时空索引与计算引擎，相比传统文件或旧版数据库，在执行空间查询、裁剪、分析等操作时效率更高。
* **避免默认歧义**：不指定数据源时，Agent 可能调用测试数据、过时数据或默认路径下的文件，导致处理结果不一致，甚至任务失败。明确指定是从源头控制流程质量。
* **服务发布更顺畅**：直接基于数据库发布地图服务，通常更便于后续查询、渲染和复用。

### 2.2 怎么写更合适？

**1. 在提示词中首位声明**：

在“需求描述模板”中，将数据源部分放在最前。

> **正确示例**：`“使用时空数据库中的 OSM POI 数据，统计北京市各区餐饮网点密度，并生成专题图。”`

**2. 结合空间范围控制**：

处理全球或全国数据时，建议先进行空间裁剪，只保留业务所需范围。

> **优化指令**：`“使用时空数据库中的全球地形数据，裁剪出青藏高原区域，并生成地形晕渲图发布为栅格瓦片服务。”`

**3. 保留中间结果再继续**：

对于多步骤任务，建议在提示词中明确说明保留中间结果，并基于该结果继续生成新的成果。这样可以减少重复计算，也更方便回溯和调整。

> **进阶示例**：`“使用时空数据库中的 OSM 道路数据完成缓冲区分析，保留该中间结果，再基于该结果生成专题图，并另存一个新的发布结果。”`

这样做通常更有利于保持处理过程稳定。

### 2.3 位置纠偏：先校正，再分析

当点位数据与底图对不上，或者地名对应位置不准确时，建议先做地理编码或坐标纠偏，再进入后续分析与制图流程。

**适用情况：**

* GPS 设备采集的坐标与地图底图存在偏移。
* 不同坐标系（如 GCJ-02、BD-09、WGS84）混用，导致位置不一致。
* 需要根据地名获取标准经纬度，用于地图标注或空间分析。
* 需要校验第三方数据的位置是否准确。

**推荐做法：**

1. **先提供位置线索**：尽量提供地名、地址或原始坐标中的一种。
2. **先校正再继续**：先让 Agent 判断位置是否有偏移，确认后再继续制图、分析或发布。

**提示词模板：**

> `请帮我看看这个位置准不准，并帮我纠偏。`

**示例指令：**

> `请帮我看看“武汉黄鹤楼”这个位置准不准，并帮我纠偏。`
>
> `请根据“武汉黄鹤楼”和我提供的坐标，帮我判断有没有偏移。`

**使用收益：**

先校正位置，再做分析，可以减少后续出现“点位落错位置”“缓冲区分析偏离目标”“地图标注不准”等问题。

## 三、排查与发布

### 3.1 页面异常快速修复流程

当遇到地图加载失败、分析报错等问题时，可以按下面的顺序排查：

1. **现象直述**：直接向 Agent 描述问题，如“行政区划图层无法点击查询”。
2. **提供错误信息**：打开浏览器开发者工具（F12），将 Console 面板中的红色报错信息完整复制给 Agent。
3. **尝试模型切换**：不同 AI 模型在代码生成和问题诊断上的表现会有差异，切换后再试有时会更顺利。
4. **回退至稳定版本**：如果当前版本改动后出现异常，可先回退到上一个正常版本。

### 3.2 地图服务处理

* **底图服务**：测试阶段可先使用内置密钥接入天地图等公有服务。正式项目建议申请专属密钥，以减少公共密钥限流带来的影响。
* **地形服务**：发布原始 DEM 栅格数据时，视觉效果通常较为平淡。可先生成地形晕渲图（山体阴影），再发布为栅格瓦片。
* **瓦片选型**：
  * **矢量瓦片**：适用于需要高度交互、动态样式修改、属性查询的场景。
  * **栅格瓦片（晕渲图）**：适合作为静态、高性能的背景底图。
  * **组合使用**：采用“栅格底图（地形/影像）+ 矢量业务图层”的模式，兼顾视觉效果与交互灵活性。

## 四、过程管理与反馈

### 4.1 上下文与对话管理

* **对话轮数**：平台设有单次对话轮数上限，达到后提示“继续”即可延长。对于较长任务，可结合“过程存档”分段进行。
* **自动上下文压缩**：当对话内容接近上限时，平台会自动压缩上下文，保留核心参数与命令历史。

### 4.2 反馈问题

* **截图圈选**：对于界面显示、布局等问题，可截图并使用**圈选修改**功能，指出具体位置再补充说明。
* **提交反馈**：若经上述步骤问题仍存，可通过 **主菜单 → 帮助 → 问题反馈** 提交工单，并附上操作步骤、错误日志和问题截图。
